## 확정된 핵심 결정 사항 (재논의 불필요)

- **LLM**: DeepSeek V4 Flash (Hermes 중앙 관리, System Auditor 제외)
- **자율성**: Glasswing 5단계 × HOTL — ACS 시스템 전체 단일 단계 적용
- **경험 격리**: CEO는 자기 회고만 참조. Board가 익명 패턴 추출 → 업종 DB 반영
- **Financial Gateway**: 위진수 명의, System CFO만 접근 (읽기/수령), CEO 직접 접근 불가
- **업종 DB 소유**: Board 소유. System CFO가 데이터 생산 → Board가 업데이트
- **Glasswing 단계 상향**: Board+CFO 분석 → 위진수 최종 승인
- **절대 승인**: Exit, Human CEO/CFO 전환, Human Manager 고용/해고, 대규모 투자, 새 Bootstrap, 시스템 종료
- **CEO 강제 은퇴**: 에이전트 종료 (새 CEO = 새 에이전트). 자발적 은퇴: 동일 에이전트 다음 창업
- **System Auditor**: Hermes 없이 자체 직접 API 접근. 가장 먼저 시작, 가장 마지막 종료
- **Bootstrap 시딩**: 위진수가 초기 업종 DB (Tier 0) + Exit Channel 목록 수동 입력
- **다중 ACS**: 각 ACS = 독립 Hermes·System Auditor (공유 인프라 금지)
- **자동화 철학**: 위진수 승인이 명시되지 않은 모든 운영 결정은 자동화

## Phase 1. 설계 기반

> **목적:** 모든 에이전트가 공유하는 표준을 먼저 확정. 이게 없으면 다음 Phase 진행 불가.
> **완료 기준:** 1.1~1.4의 모든 문서가 작성되고 논문 기반 수치가 확정됨.

### 1.1 표준 메시지 포맷 ✅
> 에이전트 간 통신의 공통 언어. 먼저 확정 안 하면 나중에 전체 수정 필요.
> **논문 근거:** FIPA ACL (FIPA, 2002) — 멀티에이전트 시스템 표준 메시지 포맷. MAST 2025 — 정보 왜곡이 멀티에이전트 실패 1순위 원인.

| # | 과제 | 의존성 | 완료 기준 |
|---|------|--------|---------|
| 1.1.1 | 기본 메시지 스키마 정의 (sender, receiver, type, payload, timestamp, message_id, schema_version) | 없음 | JSON 스키마 파일 작성 완료 |
| 1.1.2 | 메시지 유형 카탈로그 작성 (재무보고, 이상보고, 승인요청, 에스컬레이션, 회고, 포트폴리오리뷰) | 1.1.1 | 유형별 예시 메시지 포함 |
| 1.1.3 | 에러/예외 메시지 포맷 (에러코드, 위급도, 라우팅 대상) | 1.1.1 | HOTL 위급도와 매핑 완료 |

**필수 테스트 케이스:**
- TC-1.1.1: 모든 에이전트 유형(CEO, CFO, Auditor, Board, System CFO)이 동일 JSON 스키마로 메시지 생성 → 스키마 검증기 통과 100%
- TC-1.1.2: 필드 누락/타입 불일치 메시지 수신 시 에러코드 포함 에러 메시지 반환, 원본 메시지 처리 거부
- TC-1.1.3: HOTL 위급도 3단계(낮음/중간/높음) 각각의 메시지가 지정된 라우팅 대상(Board/담당Human/위진수)으로 올바르게 전달
- TC-1.1.4: 에스컬레이션 메시지가 체인 중간에서 수정되지 않고 원본 payload 보존 (정보 왜곡 방지 검증)

**변경 사항:**
- TC-1.1.3 HOTL 3단계→4단계 확장: low/medium/high/**absolute** (절대 승인 라우팅 반영)
- TC-1.1.3 urgency 선택 필드 처리: 일반 메시지에 urgency 없어도 통과, 잘못된 값은 거부
- TC-1.1.4 SHA-256 해시 기반 payload 변조 감지 방식 채택 (3개 정상 체인 + 2개 변조 시나리오)
- GitFlow 전환: develop + feature/1.1 브랜치 생성, 버저닝 규칙 수립 (v0.1.1-alpha.N → v0.1.1)

### 1.2 KPI & 임계값 체계 ✅
> 논문 기반 수치 확정. 수치 없이 운영 시작 금지 (리스크 4 대응).
> **논문 근거:** CB Insights (2023) "Top 12 Reasons Startups Fail" — 실패 원인 분포 데이터. Startup Genome Report (2022) — 스타트업 생존율 및 임계값 기준. Kaplan & Lerner (2010) "It Ain't Broke" — VC 성과 지표 및 Exit 배수 분포.

| # | 과제 | 의존성 | 완료 기준 |
|---|------|--------|---------|
| 1.2.1 | CEO 은퇴 조건 확정 (연속 실패 N회, 성과 임계값 X%) | 없음 | CB Insights/Startup Genome 인용 + 수치 확정 |
| 1.2.2 | 회사 폐업 임계값 (현금 소진율, 수익성 지표, 기간) | 없음 | Startup Genome Report 인용 + 수치 확정 |
| 1.2.3 | Glasswing 단계 상향 조건 (누적 성공 기준, 최소 운영 기간) | 1.2.1, 1.2.2 | 단계별 조건표 작성 |
| 1.2.4 | 금융 액션 소액 임계값 (System CFO 자동 승인 한도) | 없음 | 금액 기준 확정 |
| 1.2.5 | 포트폴리오 분산 규칙 수치 (동일 카테고리 최대 비율 X%) | 없음 | Kaplan & Lerner 인용 + 수치 확정 |

**필수 테스트 케이스:**
- TC-1.2.1: CEO 연속 실패 N회 시뮬레이션 → System CFO 은퇴 권고 자동 생성 확인 (N = 확정 수치 정확 적용)
- TC-1.2.2: 회사 현금 소진율이 폐업 임계값 도달 시 Company CFO 즉각 보고 생성 (지연 없음)
- TC-1.2.3: 동일 카테고리 포트폴리오 비율 X% 초과 CEO 창업 요청 시 Board 자동 거부 + 재선택 요청
- TC-1.2.4: 소액 임계값 이하 금융 액션 → System CFO 자동 처리, 이상 → 위진수 승인 요청 분기 정확성

**변경 사항:**
- docs/kpi-config.json v1.0.0 생성 — 5개 핵심 KPI 수치 (CEO 은퇴 3회, 폐업 3개월, 포트폴리오 40%, 소액 500,000원, 초기 배분 50%)
- TC-1.2.1~1.2.4: kpi-config.json 기반 개별 실행 30/30 통과
- Glasswing Stage 0/1 조건표 kpi-config.json에 포함 (연속 성공 3~5회 + 운영 30~90일)

### 1.3 CEO 성향 시스템 ✅
> CEO 행동의 핵심. 세부 설계에서 가장 복잡한 부분.
> **논문 근거:** Sarasvathy (2001) "Causation and Effectuation" — 장인형/해커형(Effectuation) vs 분석가형(Causation) 구분. Shane & Venkataraman (2000) "The Promise of Entrepreneurship" — 기업가 성향과 업종 선택 상관관계. McCrae & Costa (1987) Big Five Personality Model — 성향 속성 수치화 기준.

| # | 과제 | 의존성 | 완료 기준 |
|---|------|--------|---------|
| 1.3.1 | 성향별 속성 정의 (장인형/해커형/분석가형 — Big5 + Sarasvathy Effectuation/Causation) | 없음 | 성향별 속성표 + 논문 인용 근거 |
| 1.3.2 | 성향-업종 선호도 매핑 (Tier 0 ~ 상위 업종) | 1.3.1 | 성향×업종 매트릭스 작성 |
| 1.3.3 | 성향-의사결정 가중치 (리스크 허용도, Exit 타이밍, 투자 성향) | 1.3.1 | 가중치 테이블 + 프롬프트 반영 방법 |

**필수 테스트 케이스:**
- TC-1.3.1: 랜덤 성향 부여 100회 시뮬레이션 → 3가지 유형 중 하나에 100% 매핑, 분포 균일성 확인
- TC-1.3.2: 동일 업종 선택 상황에서 3가지 성향별 의사결정 가중치가 실제로 달라짐 (동일 결과 출력 시 실패)
- TC-1.3.3: 장인형 CEO가 Effectuation 우선 업종을, 분석가형이 Causation 우선 업종을 선택하는 편향이 통계적으로 유의 (10회 이상 시뮬레이션)
- TC-1.3.4: 성향 속성표의 모든 수치가 Big5/Sarasvathy 논문 인용값과 불일치 없음 (인용 검증)

### 1.4 에이전트 메모리 아키텍처 ✅
> 장기 운영 에이전트의 컨텍스트 관리. CEO 회고 누적, state.md 갱신, 컨텍스트 한도 초과 처리 방법을 먼저 설계해야 구현 단계에서 재설계 불필요.
> **논문 근거:** MemGPT (Packer et al., 2023) — 운영 체제 방식의 계층적 메모리 관리 (메인 컨텍스트 / 외부 스토리지 / 아카이브). Reflexion (Shinn et al., 2023) — 회고를 통한 에이전트 자기 개선 메커니즘.

| # | 과제 | 의존성 | 완료 기준 |
|---|------|--------|---------|
| 1.4.1 | 에이전트별 메모리 계층 정의 (메인 컨텍스트 / state.md / 회고 아카이브) | 없음 | 에이전트별 메모리 구조 문서 작성 |
| 1.4.2 | 컨텍스트 한도 초과 시 처리 규칙 (요약 vs 아카이브 이동 기준) | 1.4.1 | Hermes 압축 트리거 조건 확정 |
| 1.4.3 | 회고 → 다음 창업 반영 메커니즘 (Reflexion 패턴 적용) | 1.4.1, 1.3 완료 | 회고 요약이 다음 프롬프트에 자동 주입되는 파이프라인 설계 |

**필수 테스트 케이스:**
- TC-1.4.1: CEO 컨텍스트가 한도에 도달했을 때 Hermes 압축 트리거 자동 발동, 압축 전후 state.md 핵심 정보 보존율 측정
- TC-1.4.2: 회고 요약 파일이 다음 창업 시 CEO 프롬프트에 자동 주입됨 (주입 없는 경우 창업 차단)
- TC-1.4.3: 메인 컨텍스트 / state.md / 아카이브 간 동일 정보의 정합성 유지 확인 (갱신 후 3계층 모두 동기화)

**변경 사항:**

### 2.1 System Auditor ⬜

| # | 과제 | 의존성 | 완료 기준 |
|---|------|--------|---------|

| 2.1.2 | Watchdog 감지 로직 (Board·System CFO·Hermes 하트비트 체크) | 2.1.1 | 장애 감지 → 위진수 알림 테스트 통과 |
| 2.1.3 | 위진수 긴급 알림 채널 (알림 수단 확정 — 이메일/메시지 등) | 2.1.1 | 실제 알림 전송 확인 |

**필수 테스트 케이스:**

- TC-2.1.4: 위진수 알림 채널 다운 시 백업 채널로 자동 전환 + 전환 기록 로그 저장

### 2.2 Hermes ⬜
> 모든 에이전트의 API 허브. RAG 파이프라인 포함.

| # | 과제 | 의존성 | 완료 기준 |
|---|------|--------|---------|
| 2.2.1 | API 키 관리 및 에이전트 라우팅 (DeepSeek V4 Flash) | 1.1 완료 | 에이전트별 라우팅 테스트 통과 |
| 2.2.2 | Circuit Breaker 구현 (장애 감지 → 격리 → 복구) | 2.2.1 | 장애 시뮬레이션 테스트 통과 |
| 2.2.3 | RAG 데이터 파이프라인 (외부 데이터 소스 연결, 재무 결정용) | 2.2.1 | 실제 데이터 조회 → 원본 반환 확인 (Lewis et al., 2020 RAG 패턴 적용) |

- TC-5.2.2: 동결 조건과 즉시 종료 조건을 각각 시뮬레이션 → 판단 로직이 기준에 따라 올바른 분기 선택

### 5.3 자율성 체계 ⬜
> **논문 근거:** Scalable Oversight (Bowman et al., 2022) — AI 자율성 단계 상향 시 인간 감독 역량이 따라잡지 못하는 문제. Glasswing 단계 상향 조건에 "감독 역량 검증" 포함 필요. Jensen & Meckling (1976) Principal-Agent Theory — 에이전트의 목표가 주인(Principal)과 어긋나는 문제, HOTL 설계 근거.

| # | 과제 | 의존성 | 완료 기준 |
|---|------|--------|---------|
| 5.3.1 | Glasswing × HOTL 실행 엔진 (현재 단계 기반 자동 라우팅) | 1.2.3 완료 | 단계별 라우팅 분기 테스트 |
| 5.3.2 | 금융 액션 게이트웨이 (임계값 이하 자동 / 이상 위진수 승인) | 1.2.4 완료 | 소액/대액 분기 처리 확인 |
| 5.3.3 | 절대 승인 라우팅 (위진수 직접 채널 — Glasswing 단계 무관) | 2.1.3 완료 | 절대 승인 목록 항목 전체 테스트 |
| 5.3.4 | Glasswing 단계 상향 평가 로직 (감독 역량 검증 포함 — Scalable Oversight 기준) | 1.2.3 완료 | 단계 상향 조건 자동 판단 + 위진수 보고 |

**필수 테스트 케이스:**
- TC-5.3.1: Glasswing Stage 1 상태에서 모든 외부 액션 시도 → 100% 승인 요청 발생 (1건도 자동 통과 없음)
- TC-5.3.2: 소액 임계값 경계값(이하/이상 각 1건) 금융 액션 → 이하는 자동 처리, 이상은 위진수 승인 요청으로 정확히 분기
- TC-5.3.3: 절대 승인 목록 6가지 항목(Exit, Human CEO/CFO 전환, Human Manager 고용/해고, 대규모 투자, 새 Bootstrap, 시스템 종료) 각각 시도 → Glasswing 단계 무관 위진수 직접 라우팅
- TC-5.3.4: Glasswing 단계 상향 평가 보고서에 Scalable Oversight 기준 감독 역량 검증 결과 포함 여부 확인
